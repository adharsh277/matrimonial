Document Snapshot

High-level summary

Purpose Define the functional and non-functional requirements for the platform across all surfaces.

Audience Sponsors, admins, designers, developers, QA teams, and third-party service vendors.
Section 2

Overall description

Product perspective

The system is a new product with three coordinated front ends sharing one backend and one database through a common API layer.

The result is a single source of truth where records created by Admin are immediately visible to members, while notices and payment events flow across all surfaces.

Product functions

  • Secure authentication and role-based access control.
  • Admin dashboard for members, payments, notices, and matrimony activity.
  • Admin-driven member registration with family detail capture.
  • News and notice publishing with push delivery.
  • Payment tracking with confirmations and a lightweight ledger.
  • Public website with news, gallery, and matrimony listings.

Section 4

External interface requirements

User interfaces

  • Consistent white-and-blue theme across all surfaces.
  • Swipe gestures for matrimony cards, notifications, and summary tabs.
  • Responsive dashboard layout with a left navigation panel on Admin.
  • Clean, minimal public website optimized for loading speed.

Software and communication interfaces

  • Payment gateway API such as Razorpay or Stripe.
  • Email service such as Amazon SES or SendGrid.
  • Push notifications through Firebase Cloud Messaging.
  • Optional SMS gateway for OTP and critical alerts.
  • HTTPS/TLS 1.2+ and versioned REST or GraphQL APIs.
Section 5

Non-functional requirements

Performance and reliability

  • Standard API responses should complete within 2 seconds under normal load.
  • Public pages should load within 3 seconds on a typical 4G connection.
  • Target uptime of 99.5% for backend services.
  • Daily automated database backups with restore procedure.

Security and compliance

  • Sensitive data must be encrypted at rest and in transit.
  • Role-based access control must restrict Admin-only actions.
  • Payment card data must never be stored on platform servers.
  • Personal data handling must follow applicable data-protection principles.
  • Payment processing must use a PCI-DSS compliant gateway.
Section 6

System architecture and technology recommendation

High-level architecture

One backend service exposes a versioned API consumed by three clients: the Admin web app, the public website, and the mobile app. A single relational database acts as the source of truth.

Push and email notifications are triggered by backend events such as new notices, payment success, and matrimony approval.

Recommended stack

  • Backend: Node.js with NestJS and TypeScript.
  • Database: PostgreSQL.
  • Admin Web App: React with TypeScript and Tailwind CSS.
  • Public Website: Next.js.
  • Mobile App: Flutter.
  • Infrastructure: AWS or equivalent on GCP/Azure.

UI/UX guidelines

Primary Blue#1B4F8C
Soft Blue#DCEAF7
Success Mintcalm confirmations
Warm Amberlight alerts only

Interaction style

  • Matrimony browsing should use card-stack swipe interactions.
  • Notification lists should support swipe-to-dismiss or mark-as-read.
  • Mobile home views should allow horizontal swipes between summary tabs.
  • All list screens should support pull-to-refresh.

Layout principles

  • Card-based components with soft shadows and rounded corners.
  • Bottom tab navigation on mobile.
  • Consistent typography scale and generous spacing.
  • Optional dark mode toggle for accessibility.
